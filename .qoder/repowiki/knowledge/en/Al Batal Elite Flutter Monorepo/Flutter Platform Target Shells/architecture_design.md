Each subdirectory is a self-contained Flutter platform target generated by `flutter create`, with no cross-target dependencies — dependency direction flows from Dart code into these native hosts only.

- Android (`android/`): Gradle/Kotlin project; `MainActivity.kt` extends `FlutterActivity` and relies on `GeneratedPluginRegistrant.java` to register plugins at launch.
- iOS (`ios/`): Xcode workspace with `Runner/AppDelegate.swift` implementing `FlutterAppDelegate` plus `FlutterImplicitEngineDelegate`; plugin registration happens in `didInitializeImplicitFlutterEngine` via `GeneratedPluginRegistrant`.
- Linux (`linux/`): CMake-based GTK host; `main.cc` constructs a `MyApplication` (GLib/GTK) which creates the Flutter view through the embedded engine.
- macOS (`macos/`): Cocoa/Xcode project mirroring iOS structure; `AppDelegate.swift` + `MainFlutterWindow.swift` bootstrap the Flutter window.
- Windows (`windows/`): CMake + Win32 host; `main.cpp` initializes COM, sets up a `DartProject` pointing at `data/`, and drives a `FlutterWindow` backed by `Win32Window`.
- Web (`web/`): Static entrypoint `index.html` referencing `$FLUTTER_BASE_HREF`, `manifest.json`, PWA icons, and `flutter_bootstrap.js` produced by the Flutter web build.

All platforms share the same application name `albatal_store` (visible in Windows window title, iOS/macOS Info.plist, Android package, and web `<title>`). Build manifests (`build.gradle.kts`, `CMakeLists.txt`, `project.pbxproj`, `Info.plist`, `AndroidManifest.xml`) are the only integration points between Dart and native code.
