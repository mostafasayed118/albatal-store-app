Flat collection of deployment utilities split by language:
- PowerShell entry points (`run_migrations.ps1`, `combine_migrations.ps1`) read `supabase/migrations/*.sql` files sorted by their `NNN_` prefix, then either concatenate them into `scripts/run_all_migrations.sql` (File mode) or execute each via the Supabase Management API `POST /v1/projects/{ref}/database/query` using `SUPABASE_PROJECT_REF` + `SUPABASE_SERVICE_ROLE_KEY` (API mode).
- Bash/Windows batch (`deploy-staging.sh`, `deploy-staging.bat`, `deploy-staging.ps1`) wrap the Supabase CLI: link a project from `.env.staging`, push migrations with `supabase db push --linked`, then deploy only the whitelisted active Edge Functions (`checkout`, `paymob-initiate`, `paymob-callback`, `cancel-expired-orders`, `send-order-notification`) while explicitly excluding deprecated ones.
- Standalone SQL seeds (`drop_everything.sql`, `run_all_migrations_template.sql`, `run_new_migrations_only.sql`, `verify_schema.sql`) complement the combined artifact for one-shot setup, selective re-run, and schema verification.
Dependency direction is one-way: scripts depend on `supabase/migrations/`; the generated `run_all_migrations.sql` is an output artifact consumed by the SQL Editor.
